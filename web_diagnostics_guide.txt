ДИАГНОСТИКА ПРОБЛЕМ С ЗАГРУЗКОЙ САЙТА
======================================

Этап 1: DNS-запрос
------------------
Что это: браузер переводит доменное имя (sfmshop.com) в IP-адрес.

Как проверить:
  nslookup sfmshop.com
  dig sfmshop.com

Какие проблемы могут быть:
  - DNS-сервер не отвечает
  - Домен не зарегистрирован или истёк
  - Неправильно настроены DNS-записи
  - Кэш старой записи на стороне клиента

Как решить:
  - Сменить DNS-сервер на публичный (8.8.8.8 от Google или 1.1.1.1 от Cloudflare)
  - Очистить DNS-кэш: ipconfig /flushdns (Windows) или sudo systemd-resolve --flush-caches (Linux)
  - Проверить и обновить DNS-записи у регистратора домена


Этап 2: TCP-соединение
----------------------
Что это: браузер устанавливает соединение с сервером по IP-адресу.

Как проверить:
  ping sfmshop.com
  telnet sfmshop.com 80
  telnet sfmshop.com 443

Какие проблемы могут быть:
  - Сервер не отвечает (упал или перегружен)
  - Порт закрыт файрволом
  - Высокий ping — сервер далеко или перегружена сеть
  - Пакеты теряются

Как решить:
  - Проверить статус сервера у хостинг-провайдера
  - Открыть нужные порты (80, 443) в настройках файрвола
  - Перезапустить сервер если не отвечает


Этап 3: HTTP-запрос
-------------------
Что это: браузер отправляет запрос на получение страницы.

Как проверить:
  curl -v https://sfmshop.com
  Инструменты разработчика браузера → вкладка Network → выбрать запрос → Headers

Какие проблемы могут быть:
  - Неправильные заголовки запроса
  - Проблемы с SSL-сертификатом (сайт по https недоступен)
  - Редиректы зациклились
  - Сервер отклоняет запрос

Как решить:
  - Проверить срок действия SSL-сертификата и обновить его
  - Найти и убрать лишние редиректы в настройках сервера
  - Проверить заголовки через curl -I https://sfmshop.com


Этап 4: Обработка на сервере
-----------------------------
Что это: сервер получает запрос, выполняет код, обращается к БД и формирует ответ.

Как проверить:
  - Открыть логи сервера: /var/log/nginx/error.log или /var/log/apache2/error.log
  - Посмотреть время ответа в инструментах разработчика → Network → столбец Time
  - Проверить загрузку сервера: top, htop

Какие проблемы могут быть:
  - Ошибка в коде приложения (500 Internal Server Error)
  - Медленные запросы к БД
  - Сервер перегружен (мало памяти или CPU)
  - Зависшие процессы

Как решить:
  - Найти ошибку в логах и исправить код
  - Оптимизировать медленные SQL-запросы (добавить индексы)
  - Перезапустить зависшие процессы
  - Увеличить ресурсы сервера


Этап 5: HTTP-ответ
------------------
Что это: сервер отправляет ответ обратно браузеру.

Как проверить:
  curl -I https://sfmshop.com  — посмотреть статус и заголовки ответа
  Инструменты разработчика → Network → Status, Size, Headers

Основные статус-коды:
  200 — всё хорошо
  301/302 — редирект
  404 — страница не найдена
  500 — ошибка на сервере
  503 — сервер временно недоступен

Какие проблемы могут быть:
  - Большой размер ответа (тяжёлые картинки, нет сжатия)
  - Неправильные заголовки кэширования
  - Ошибочный статус-код

Как решить:
  - Включить сжатие gzip на сервере
  - Оптимизировать изображения
  - Настроить правильные заголовки Cache-Control


Этап 6: Рендеринг
-----------------
Что это: браузер строит страницу из полученного HTML, CSS и JavaScript.

Как проверить:
  Инструменты разработчика браузера:
  - вкладка Performance → записать загрузку → найти узкие места
  - вкладка Lighthouse → запустить аудит → получить оценку и рекомендации
  - вкладка Network → посмотреть какие файлы грузятся дольше всего

Какие проблемы могут быть:
  - Тяжёлые JS-файлы блокируют отрисовку страницы
  - Много CSS замедляет построение стилей
  - Неоптимизированные картинки
  - Много запросов к серверу при загрузке

Как решить:
  - Подключать JS с атрибутом defer чтобы не блокировал рендеринг
  - Минифицировать CSS и JS файлы
  - Использовать ленивую загрузку картинок (lazy loading)
  - Объединять несколько файлов в один чтобы сократить количество запросов
